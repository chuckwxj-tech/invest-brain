## 原文

@francoisfleuret I don't think it would be very hard, but there are nuances. Here is some back-of-the-napkin thoughts:
Assuming we're talking about standard hardware (not Nvidia CPU<>GPU NVLink bus, which requires Grace processor), you'd need a processor that can handle that many PCIe lanes. The best current AMD CPU has 128 lanes of PCIe 5.
- Each NVMe needs 4-lanes. So 16x4 = 64 lanes.
- Single GPU needs 16-lanes
So 80 lanes total for the M.2 & GPU. That shoud leave plenty for the rest of the system... doable with 1 GPU
That's just a single GPU though. If you wanted to have multiple GPUs in a system, each adds 16x lanes.
So on an 8xGPU system, that's 128-lanes just for the GPUs. In order to accommodate that, you now need to have multiple processors. This introduces a new bottleneck of inter-processor communication.
With 2 CPUs, you can theoretically have 16xM.2 and 8xGPUs, but you would need to have GPUDirect-Storage and the data would need to be properly sharded across the drives and read into the GPU on the appropriate NUMA. This is more of a software & driver issue than a hardware limitation.
Since the GPUs only have ~141GB-288GB HMB3e each, it's not a massive amount of data to populate and a typical server would have 2TB-4TB of memory.
So, in theory it's possible, but you don't see it very often because it's just easier to give the node more memory (especially with new CXL standards) and just move the data into memory.
Again, just thoughts off the cuff here. There are more advanced technology and things that can be used.
